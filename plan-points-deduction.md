# 积分扣除功能开发计划

## 功能概述
开发一个积分扣除功能，允许家长在孩子表现不好时扣除积分，作为负面行为的惩罚机制。

## 背景分析
- 现有系统已有积分增加功能（`POST /api/points`）
- 用户模型已有 `totalPoints` 和 `availablePoints` 字段
- 已有账本系统（ledger）记录积分收入和支出
- 需要扩展支持积分扣除（负向操作）

## 功能需求

### 1. 核心功能
- [x] 家长可以通过界面扣除孩子积分
- [x] 扣除积分时需要填写原因
- [x] 支持PIN码验证（安全机制）
- [x] 扣除后实时更新孩子积分余额
- [x] 扣除记录需要持久化存储

### 2. 业务规则
- [x] 扣除积分不能超过孩子当前可用积分
- [x] 需要记录扣除操作人（家长）
- [x] 扣除原因不能为空
- [x] 支持扣除部分或全部积分

### 3. 数据模型
- [x] 扩展现有账本记录，支持"扣除"类型
- [x] 复用Order模型，添加type和deductedBy字段
- [x] 记录字段：孩子ID、扣除积分、原因、操作人、时间

### 4. 用户界面
- [x] 在家长端添加"扣除积分"按钮/入口
- [x] 扣除积分弹窗/表单
  - [x] 积分输入框（带验证）
  - [x] 原因输入框（必填）
  - [x] PIN码验证
  - [x] 确认按钮
- [x] 扣除成功提示
- [x] 在账本中显示扣除记录

### 5. API接口
- [x] `POST /api/points/deduct` - 扣除积分接口
  - 参数：childId, points, reason, pin
  - 返回：更新后的积分信息
- [x] 更新 `GET /api/ledger` - 支持查询扣除记录

### 6. 安全考虑
- [x] PIN码验证
- [x] 家长权限验证
- [x] 扣除金额验证（不能为负数，不能超过可用积分）
- [x] 操作日志记录

## 技术实现方案

### 方案选择
**方案A：扩展现有积分调整API**
- 修改现有 `POST /api/points` 支持负数
- 优点：简单，复用现有代码
- 缺点：不够明确，安全性较低

**方案B：创建独立的扣除API（推荐）**
- 新建 `POST /api/points/deduct` 接口
- 独立的权限和验证逻辑
- 优点：职责清晰，安全性高，易于维护
- 缺点：需要新建接口

### 采用方案B

## API设计

### POST /api/points/deduct

**请求参数：**
```json
{
  "childId": "string",
  "points": "number", // 正数，表示要扣除的积分
  "reason": "string",
  "pin": "string"
}
```

**成功响应：**
```json
{
  "success": true,
  "data": {
    "child": {
      "id": "string",
      "nickname": "string",
      "avatar": "string",
      "totalPoints": "number",
      "availablePoints": "number"
    },
    "deductedPoints": "number",
    "reason": "string"
  }
}
```

**错误响应：**
```json
{
  "success": false,
  "message": "string"
}
```

## 数据库设计

### 选项1：复用现有Order模型（推荐）
将扣除视为一种"负向订单"，复用Order模型，添加type字段区分：
- `type: 'reward'` - 兑换奖励
- `type: 'deduction'` - 积分扣除

### 选项2：创建新模型DeductionRecord
```typescript
interface IDeductionRecord {
  childId: ObjectId;
  points: number;
  reason: string;
  deductedBy: ObjectId; // 家长ID
  createdAt: Date;
}
```

**采用选项1**（复用Order模型，最小化改动）
