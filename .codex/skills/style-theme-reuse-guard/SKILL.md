# style-theme-reuse-guard

在进行功能开发、修改或重构时，默认先做约束检查，再开始实现。

## 适用场景

- 新增功能
- 修改已有功能
- 替换或新增页面/组件
- 调整样式、交互、布局或主题表现

## 页面布局规则

当页面采用“左侧菜单 + 右侧内容”的结构时，默认遵循以下布局：

- 左侧为菜单区域
- 右上方为页面标题
- 右下方为主体内容区域
- 主体内容区域内容过高时，主体区域内部显示滚动条
- 主体内容区域顶部优先放查询条件，如果没有查询条件，再放其他辅助操作区
- 主体内容区域下方放主要内容

## 工作规则

1. 先检查当前代码风格、目录结构、命名方式和同类实现。
2. 先找现成组件、现成页面片段、现成工具函数。
3. 能复用就复用，只有确认现有能力不够时才新增组件。
4. 新增内容必须尽量贴合仓库已有风格，不要引入突兀的新范式。
5. 如果项目支持多个主题，开发时必须逐个主题检查和适配。
6. 不要只验证默认主题，至少确认每个主题下的视觉和交互都可用。

## 开发检查清单

- 是否已经阅读同类模块的实现？
- 是否已经确认有可复用组件？
- 如果新增了组件，是否真的无法复用现有组件？
- 新代码是否沿用了现有命名、结构和状态管理方式？
- 是否已对所有主题完成适配？
- 切换主题后，颜色、对比度、边距、阴影和状态是否都正常？
- 每次样式调整后，按钮背景色与文字对比度是否正确？
- 左侧菜单、页面标题和主体内容区域的布局是否符合约定？
- 主体内容区域是否只在内部滚动，而不是整页滚动？
- 查询条件是否位于主体顶部，主要内容是否位于其下方？

## 适配原则

- 优先使用主题变量、设计令牌或已有的主题机制。
- 不要把颜色、间距、阴影写死在局部逻辑里，除非仓库本来就是这么做的。
- 如果一个改动只适配了单一主题，就视为未完成。

## 输出要求

实现时优先给出以下结论：

- 复用了哪些现有组件或模式
- 为什么没有新增组件，或者为什么必须新增
- 已经检查了哪些主题
- 每个主题下需要特别留意的差异
